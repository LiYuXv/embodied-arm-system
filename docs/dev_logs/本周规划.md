## 第 1 天：完善 `embodied_motion`，把运动层收稳

目标：运动执行层不只是能 `home/observe`，而是具备可扩展的正式接口。

要完成：

```text
1. 给 embodied_motion 增加更多安全位姿
   - home
   - observe
   - ready
   - pre_pick

2. 完善 /motion/go_named_pose
   - 检查 action server 是否在线
   - 返回更清楚的成功/失败信息
   - 避免重复发送目标

3. 初步研究 gripper_controller
   - 查 gripper_controller 控制的关节
   - 确认夹爪能否通过 FollowJointTrajectory 控制
```

验收标准：

```text
ros2 service call /motion/go_named_pose ... home 成功
ros2 service call /motion/go_named_pose ... observe 成功
ros2 service call /motion/go_named_pose ... ready 成功
```

当天记录重点：

```text
motion_config.yaml 新增哪些位姿
gripper_controller 是否能控制
是否遇到控制器拒绝目标的问题
```

## 第 2 天：开始 `embodied_language`，实现中文输入到任务指令

目标：让系统从“手动 service 调用”变成“中文指令触发”。

要完成：

```text
1. 创建 language_node.py
2. 支持终端输入中文指令
3. 规则解析基础动作
   - 回到初始位置 → home
   - 移动到观察位置 → observe
   - 准备抓取 → pre_pick
4. 输出 TaskCommand
5. 调用 /motion/go_named_pose 服务
```

第一版先支持：

```text
回到初始位置
回家
移动到观察位置
去观察位置
准备抓取
```

验收标准：

```text
ros2 run embodied_language language_node

请输入指令：移动到观察位置
→ language_node 解析 action
→ 调用 /motion/go_named_pose
→ 机械臂移动到 observe
```

当天结果应该是：

```text
中文输入 → embodied_language → /motion/go_named_pose → 机械臂执行
```

这就是第一个“语言到执行”的正式闭环。

## 第 3 天：加入 `embodied_task`，把语言和运动解耦

目标：不要让 language 直接控制 motion，而是走正式系统结构。

现在第 2 天可能会让 language 直接调用 motion，这是为了先打通。第 3 天要改成：

```text
embodied_language
  ↓ 发布 TaskCommand
embodied_task
  ↓ 解析任务流程
embodied_motion
  ↓ 执行
```

要完成：

```text
1. language_node 发布 /task_command
2. task_manager_node 订阅 /task_command
3. task_manager_node 根据 action 调用 motion 服务
4. language_node 不再直接控制机械臂
```

验收标准：

```text
输入：回到初始位置
language_node 发布 TaskCommand
task_manager_node 收到 TaskCommand
task_manager_node 调用 /motion/go_named_pose
机械臂执行 home
```

当天结束后，你的架构会更像最终系统：

```text
语言模块只负责理解
任务模块负责调度
运动模块负责执行
```

## 第 5 天：补仿真场景方案，决定 RGB-D 相机怎么进来

目标：确定完整仿真感知环境怎么搭。

你现在的 EDULITE_A3 环境只有 MoveIt2/RViz，没有相机 topic。所以第 5 天要解决：

```text
仿真相机从哪里来？
桌面和目标物体在哪里？
RGB-D topic 怎么发布？
```

建议当天完成两件事：

```text
1. 阅读 EDULITE_A3 是否已有 Gazebo / sim 相关 launch
2. 如果没有，就确定后续方案：
   - MoveIt2/RViz 继续作为机械臂规划执行环境
   - 另建 Gazebo / Isaac 场景用于 RGB-D 感知
   - 或先用 D435i 真实相机替代仿真相机输入
```

验收标准：

```text
写一份 docs/dev_logs/仿真视觉环境方案.md
明确：
- 当前 MoveIt2 demo 缺什么
- 后续选择 Gazebo 还是 Isaac
- RGB-D topic 名称规划
- perception_node 输入输出接口
```

这一天不一定要跑通相机，但必须把技术路线定死。
